## 1. Epic Summary

Epic 1 delivered a configuration‑driven synthetic data generator that produces realistic food‑delivery event streams for later Spark and dashboard work:

- Core generator configuration model (`GeneratorConfig`) and shared YAML/env loader (Story 1.1).
- Two well‑defined feeds (order events and courier status) with JSON and AVRO schemas and schema‑validated outputs (Story 1.2).
- Configurable edge‑case behavior (late/out‑of‑order events, duplicates, missing steps, impossible durations, courier offline) driven entirely by configuration and compatible with downstream jobs (Story 1.3).
- Sample mode that produces small, finite, schema‑aligned batches for inspection (Story 1.4).
- Debug mode with capped entity counts and throughput to support quick, reproducible demo runs (Story 1.5).
- Curated, schema‑valid edge‑case example files and documentation explaining how those encodings should surface in Spark metrics and dashboards (Story 1.6).

From a generator perspective, Epic 1 stands up a strong technical foundation for later ingestion, windowed metrics, anomaly scores, and dashboards.

---

## 2. What Went Well (Tech & Data)

1. **Configuration as a single source of truth**
   - Generator behavior is consistently driven by `GeneratorConfig` and `config/generator.yaml`, with validation and structured error reporting.
   - This avoids hard‑coded “magic numbers” and keeps later changes to entity counts, rates, and modes low‑risk.

2. **Schema‑first feed design**
   - AVRO and JSON schemas for `order_events` and `courier_status` are clearly defined and kept in sync.
   - Tests validate that generated events conform to these schemas, reducing downstream parsing and schema‑drift risk.

3. **Rich, configurable edge‑case modeling**
   - All key streaming correctness edge cases (late/out‑of‑order, duplicates, missing steps, impossible durations, courier offline) are explicitly modeled and driven by named config fields.
   - Encodings are documented so that Spark and dashboard logic can interpret them consistently.

4. **High test coverage and tooling alignment**
   - Unit and integration tests exercise configuration validation, dual‑feed generation, JSON/AVRO serialization, sample/debug behaviors, and curated edge‑case examples.
   - Generator outputs are designed to feed PySpark Structured Streaming and Azure Event Hubs patterns described in the architecture.

5. **Operational modes aligned with teaching goals**
   - Sample and debug modes make it easy to generate small, inspectable batches and fast end‑to‑end runs without reconfiguring core logic.
   - This strongly supports the course’s educational objectives (FR32/FR33) around understanding edge cases and streaming correctness.

## 3. Challenges and Risks

1. **Complexity of configuration surface**
   - The combination of core parameters, output formats, edge‑case rates, sample mode, and debug mode creates a relatively large config surface.
   - New contributors may need guidance to avoid conflicting or confusing combinations (for example overlapping caps vs. sample sizes vs. edge‑case rates).

2. **Cognitive load around edge‑case semantics**
   - While encodings are documented, understanding how late/out‑of‑order, duplicate, missing‑step, and impossible‑duration scenarios interact can be non‑trivial.
   - There is a risk that future Spark jobs or dashboards misinterpret subtle cases if documentation and examples are not kept in lock‑step with any generator changes.

3. **Downstream pipeline still partially theoretical**
   - Generator outputs are designed for Spark/Event Hubs/Streamlit, but full end‑to‑end validation (generator → Event Hubs → Spark → Parquet → dashboard) is deferred to later epics.
   - Until that pipeline exists, some assumptions about performance, watermark behavior, and anomaly metrics remain unproven.

4. **Maintaining schema and sample alignment over time**
   - Multiple locations reference schemas and samples (AVRO files, JSON examples, Spark schemas, docs, curated edge‑case files).
   - Without disciplined change management, there is a risk of subtle drift between these artifacts.

---

## 4. Key Insights / Lessons Learned

1. **Schema‑driven design pays off early**
   - Investing early in AVRO/JSON schemas and enforcing them in tests significantly reduces uncertainty for later analytics work.
   - This makes it much easier to reason about watermarks, windowed metrics, and anomaly scores before any Spark code is written.

2. **Config‑driven edge cases are powerful but need strict guardrails**
   - Centralizing edge‑case behavior in configuration makes it easy to teach and demo scenarios, but requires clear validation and documentation to avoid ambiguous runs.
   - Having named rates and deterministic debug/sample paths is key to making edge cases reproducible and inspectable.

3. **Sample/debug modes are essential for fast feedback**
   - Being able to spin up small, finite batches or capped debug runs accelerates development and troubleshooting.
   - These modes should remain first‑class citizens and be kept in sync with any generator or schema evolution.

4. **Curated examples bridge implementation and analytics**
   - The curated JSON/AVRO edge‑case slices are an effective bridge between generator internals and downstream Spark logic.
   - Treating these examples as “living contracts” will help keep generator and analytics code aligned as the project evolves.
