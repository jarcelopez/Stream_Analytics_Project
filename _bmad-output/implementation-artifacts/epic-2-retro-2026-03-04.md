### 2. What Went Well

1. **Design note as a single semantic contract**
   - `docs/design_note.md` now acts as the authoritative description of the two feeds, their schemas, edge cases, and how they support Milestone 2 analytics.
   - It ties AVRO/JSON schemas, generator behavior, and planned metrics together so that Spark and dashboard work can rely on documentation instead of reverse‑engineering code.

2. **Run instructions aligned with architecture and PRD**
   - The `README.md` now contains a focused “Running the Milestone 1 Generator and Samples” section that matches the architecture document’s structure and terminology.
   - Sample and debug flows are documented in a way that supports FR1–FR6, FR32–FR33, and NFR5/NFR6 without requiring readers to dig into implementation details.

3. **Consistency with architecture and naming conventions**
   - Documentation consistently uses the architecture’s naming and structure rules (snake_case fields, `window_start`/`window_end`, `event_time`, package layout).
   - References to generator config, schemas, and logs are aligned with the agreed project structure, which reduces future cognitive load when implementing Spark jobs and dashboards.

4. **Clear bridge from requirements to artifacts**
   - Both stories explicitly map back to PRD functional and non‑functional requirements and to the architecture decisions.
   - This makes it easy for graders (and for “future you”) to see how documentation supports the larger pipeline goals.

---

### 3. Challenges and Growth Areas

1. **Documentation staying in sync with fast‑evolving code**
   - As the generator and schemas evolve, there is a real risk that `docs/design_note.md` and README instructions drift from reality.
   - The more edge cases and configuration options are added, the harder it becomes to keep examples and descriptions perfectly accurate.

2. **High information density for new readers**
   - The design note and PRD together contain a lot of detail; intermediate‑skill readers may need guidance on where to start and what to skim.
   - Without a short, curated “on‑ramp,” new contributors can still feel overwhelmed despite good documentation.

3. **Limited automation around validation**
   - While Epic 1’s retrospective recommended schema and sample alignment tests, Epic 2 itself did not add new automated checks to keep docs and artifacts in sync.
   - There is still an opportunity to build lightweight CI checks that assert, for example, that documented examples validate against current schemas.

However, automated validation and explicit configuration‑profile definitions are still pending. Those will be more impactful once Spark ingestion and dashboard pieces exist (Epic 3 and 4).

### 5. Next Epic Preview (Epic 3 – Stream, Process, and Store Analytics‑Ready Data)

Epic 3 will take the documented feeds and turn them into a live streaming analytics pipeline:

- **Title and objective:** Stream events into Azure Event Hubs, process them with Spark Structured Streaming, and write curated Parquet metrics.
- **Planned capabilities:** Event Hubs publishing, ingestion and validation jobs with error sinks, event‑time semantics and watermarks, windowed KPIs, stateful/anomaly metrics, and Parquet outputs for inspection.
- **Dependencies on Epic 2 work:**
  - Spark schemas and parsing logic will rely directly on the contracts described in `docs/design_note.md`.
  - Debug/sample flows from the README will be the primary way to generate controlled inputs for ingestion and KPI/anomaly testing.

Readiness‑wise, Epic 2 leaves documentation in a strong enough state that Epic 3 can proceed without first “discovering” intent from generator code. The main risk is keeping that alignment as both sides evolve.
