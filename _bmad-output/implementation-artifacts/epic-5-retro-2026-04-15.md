## What Went Well

1. Epic objective was achieved end-to-end: the dashboard now controls demo lifecycle (`Start Demo`, `Stop Demo`, `Reset Demo`) without CLI juggling.
2. Story 5.1 introduced a solid orchestration foundation with idempotent start/stop/reset behavior and explicit status artifact contracts.
3. Story 5.2 improved operational visibility with explicit pipeline status badges and `last_batch_ts` observability, including degraded-state handling.
4. Story 5.3 turned implementation behavior into a reproducible evaluator flow with a dedicated runbook and docs validation tests.
5. Quality discipline was strong: review findings were addressed in-story, and targeted plus full-suite regression runs were documented.

## Challenges and Friction Points

1. Early Story 5.1 behavior exposed startup edge cases (partial startup cleanup and mixed run-state ambiguity) that required follow-up hardening.
2. Story 5.2 needed a second pass to align status refresh semantics with heartbeat approximation behavior while preserving user trust in status signals.
3. Documentation accuracy required deliberate parity work between README and dedicated runbook, plus stronger docs tests to prevent drift.
4. Story artifacts initially included minor file-list tracking inconsistencies and were corrected after review.

## Root Causes (System-Level)

- Operational state management crosses UI, orchestration, and runtime status artifacts; small contract ambiguities can quickly propagate into user-facing confusion.
- Review and test feedback loops were essential to validate behavior under failure/degraded paths, not just happy-path control flows.
- Reproducibility is a system property, not only a documentation concern; docs and implementation must evolve together to stay trustworthy.

## Key Insights and Lessons Learned

1. Lifecycle controls must be idempotent and rollback-safe by default to avoid orphaned or duplicate processes in demo environments.
2. Status semantics should distinguish healthy, degraded, and failed paths clearly; stale telemetry alone is not always an error signal.
3. Story completion quality improves when AC-to-test traceability is explicit and continuously validated.
4. Documentation tests are useful guardrails for evaluator-facing reliability, especially when runbook behavior spans multiple components.
