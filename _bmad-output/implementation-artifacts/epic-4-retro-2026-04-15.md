## What Went Well

1. Dashboard foundation landed cleanly in Story 4.1 with a clear entrypoint, typed configuration, cache-aware loading, and explicit UX states for loading/empty/error.
2. Filtering in Story 4.2 became robust after review hardening:
   - Type-safe matching for `zone_id` and `restaurant_id`
   - Bounded UTC time-window filtering
   - Better validation/warnings for invalid presets
3. Health/Anomalies capabilities in Story 4.3 delivered the intended analytical view with thresholding and top-N ranking behavior aligned to acceptance criteria.
4. Test quality improved across the epic and final dashboard-focused suites were validated successfully.
5. Story artifacts show a strong review-feedback loop, including explicit follow-up tasks and closure evidence.

## Challenges and Friction Points

1. Story 4.2 initially had multiple high/medium review findings (filter type mismatch, brittle preset handling, and window-boundary issues).
2. Some story checklist claims were initially ahead of implemented test coverage and needed synchronization.
3. Health ranking behavior required refinement (global top-N changed to per-window top-N) to fully satisfy the intended behavior.
4. Small tooling friction appeared around pytest import path assumptions and was resolved through test setup hardening.

## Root Causes (System-Level)

- Cross-cutting dashboard behavior (filters, presets, time semantics) had complexity concentrated in a single module, increasing the chance of subtle regressions.
- Acceptance criteria were clear, but edge-case enforcement required a second pass through review to validate behavior under realistic data shapes.
- Multi-page dashboard integration introduced implicit coupling that needed explicit regression coverage.

## Key Insights and Lessons Learned

1. Explicit UTC-bounded windows are essential for reliable dashboard slices; lower-bound-only filtering can silently skew insights.
2. UI selection values and dataset value types must be normalized consistently before filtering.
3. "Done" checklists need direct test evidence tied to each acceptance criterion to avoid false confidence.
4. Per-window ranking is a materially different requirement than global ranking; wording must be translated into precise assertions.
5. Lightweight warning mechanisms for invalid config values improve operability without destabilizing the app.
